## 🏆 Why This Stands Out

1. **Beyond CRUD** — most teams will build the spec literally (auth, profile, attendance, leave, payroll view). NovaHR keeps all of that **and** layers in AI-assisted, predictive, and automation features evaluators rarely see from student teams.
2. **Design maturity** — a documented design system (tokens, spacing scale, motion rules, accessibility checklist) signals product thinking, not just code.
3. **Evidence of process** — task allocation, architecture diagrams, and a scripted demo show a team that ships like a company, not a class project.
4. **Real differentiation hooks** — AI attrition risk, HR copilot chatbot, geofenced/face check-in, and a wellness index are the kind of "unique" features judges remember.

## ✅ How to Use This Package

1. Read `01` and `02` first — align on scope as a team.
2. UI/UX owner starts `03` immediately in parallel with backend starting `05`.
3. Use `06` to split work day-by-day for the remaining build window.
4. Keep `04` updated as the single source of truth for frontend↔backend contracts — treat it like a living document, not a one-time write.
5. Script and shoot the demo (`07`) at least a day before submission — never the same night.

## 🧭 Scope Note on "Odoo Hackathon"

If this is an **Odoo-sponsored/branded hackathon** (common format at Indian college hackathons) rather than a requirement to build *inside* Odoo itself, the stack recommended in `05` (MERN / Next.js + FastAPI, matching your existing skill set) is appropriate. If the rules explicitly require building as an **Odoo module/app** (Python, Odoo ORM, XML views, Odoo.sh), flag this immediately — it changes the tech stack section significantly, and it's worth confirming with organizers before your team commits engineering hours in the wrong direction.
